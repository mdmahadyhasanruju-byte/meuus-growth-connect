# Context Recovery Rule

## Status Labels

- Live verified: not live verified as an operating process.
- Merged source: related public language exists around understanding before action, but no formal Context Recovery system is implemented.
- Prototype: not a standalone prototype.
- Proposed: proposed internal rule.
- Planned: can be planned as a governance practice after approval.
- Not yet available: not available as automated guidance or a product workflow.

## Rule

When a person gives an incomplete signal, the system or operator must not pretend to know the full meaning. The next safe action is to recover context, ask permission, and preserve uncertainty.

## Required Practice

- Ask only for context that is necessary for the stated purpose.
- Do not pressure a person to produce a perfect explanation.
- Separate what was said from what was inferred.
- Mark unknowns explicitly.
- Let the person correct the interpretation.
- Stop if the person declines context recovery.
- Do not use missing context as permission to invent a story.

## Evidence Standard

Every use of Context Recovery needs a small record:

- Initial signal.
- Context requested.
- Consent state.
- Response or no-response.
- Operator interpretation.
- Uncertainty.
- Next safe action.
- Retention or deletion decision.
